## definition

A **Context-Free Grammar** (CFG) G = (V, T, S, P) where:
- V = non-terminals (variables)
- T = terminals
- S ∈ V = start symbol
- P = production rules of form **A → α** where A ∈ V, α ∈ (V ∪ T)*

"Context-free" = left-hand side is always a SINGLE non-terminal (no surrounding context required to apply a rule).

## intuition

CFGs are grammars where any non-terminal can be replaced by its right-hand side at any time, regardless of what surrounds it. This makes them powerful enough to express nested structures (like balanced parentheses or recursive function calls) while remaining efficiently parseable.

## why it exists

Regular languages cannot express balanced/nested structures or count. CFGs fill this gap:
- Can express a^n b^n (counting)
- Can express balanced parentheses (nesting)
- Can express programming language syntax (recursion)

## CFG vs Regular Grammar

| Feature | Regular Grammar | CFG |
|---------|----------------|-----|
| Rule form | A → xB or A → x | A → α (any) |
| Nesting | Cannot express | Can express |
| Examples | a*, (a+b)* | a^n b^n, palindromes |
| Recognizer | DFA | PDA |

## Derivation

**Sentential form:** any string in (V ∪ T)* reachable from S by applying rules
**Derivation:** S ⟹ α1 ⟹ α2 ⟹ ... ⟹ w (where w ∈ T*)

**Leftmost:** always replace leftmost non-terminal  
**Rightmost:** always replace rightmost non-terminal

## CFG Simplification (four methods)

1. **Substitution**: Inline one non-terminal's productions into another
2. **Remove useless symbols**: Non-generating (can't reach terminals) or non-reachable (from S)
3. **Remove ε-productions**: A → ε for non-start A; create versions with/without A in other rules
4. **Remove unit productions**: A → B; replace with B's productions in A's rules

## Chomsky Normal Form (CNF)

Every CFG has an equivalent CFG in CNF where every rule is:
- A → BC (two non-terminals), or
- A → a (single terminal)
(plus possibly S → ε)

See [[chomsky-normal-form]] for the conversion algorithm.

## applications

- **Programming language syntax** (Java, C, Python are all defined by CFGs)
- **XML/HTML parsing** (tree structure = CFG)
- **Arithmetic expressions** (operator precedence via grammar structure)
- **Natural language processing** (sentence structure)
- **Protocol specifications** (complex message formats)

## advantages

- Can express nested/recursive structures
- Efficient parsing algorithms exist (O(n³) CYK, O(n) for LL/LR grammars)
- Well-studied theory with many practical tools (yacc, bison, ANTLR)

## limitations

- Cannot count three things simultaneously (a^n b^n c^n is NOT context-free)
- Ambiguous grammars cause parsing problems
- Equivalence of two CFGs is undecidable

## misconceptions

- Not all grammars defining a language are unique — many different CFGs can generate the same CFL
- "Context-free" does NOT mean the grammar is simple — CFGs can be complex
- CFG can express a^n b^n but NOT a^n b^n c^n (need CSG for three equal-length blocks)

## examples

**L = {a^n b^n | n ≥ 1}:**
- S → aSb | ab

**Balanced parentheses:**
- S → (S) | SS | ε

**Simple arithmetic expressions:**
- E → E + T | T
- T → T * F | F
- F → (E) | id

**Equal 0s and 1s (in any order):**
- S → 1S0S | 0S1S | ε
