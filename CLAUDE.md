# knowledge vault operating system

## mission

this vault serves two purposes:

1. maximize exam performance
    
2. build a permanent knowledge archive
    

when these goals conflict:

- exam notes optimize for retrieval speed
    
- concept notes optimize for understanding
    
- never sacrifice archive quality for exam convenience
    

---

# vault structure

## 01-courses

contains lecture-derived notes.

purpose:

- preserve source material
    
- store lecture-specific context
    
- record professor-specific insights
    

not a permanent knowledge store.

---

## 02-concepts

contains atomic knowledge notes.

each note should represent exactly one concept.

examples:

- gradient descent
    
- bayes theorem
    
- tcp/ip
    
- photosynthesis
    

concept notes are the primary long-term knowledge asset.

before creating a concept note:

- search existing concepts
    
- update existing note if concept already exists
    
- never create duplicates
    

---

## 03-question-bank

contains:

- likely exam questions
    
- generated practice questions
    
- application questions
    
- comparison questions
    
- synthesis questions
    

questions should test understanding, not recall.

---

## 04-exam-notes

contains compressed revision material.

optimize for:

- speed
    
- retrieval
    
- exam navigation
    

not deep understanding.

---

## 05-cheat-sheets

contains:

- one-page summaries
    
- formulas
    
- frameworks
    
- comparison tables
    

must be usable during open-book exams.

---

## 06-project-applications

stores how concepts apply to:

- ai
    
- software engineering
    
- game development
    
- systems design
    
- real-world projects
    

---

# note generation workflow

when processing lectures:

step 1:  
extract concepts

step 2:  
update existing concept notes

step 3:  
create missing concept notes

step 4:  
generate exam questions

step 5:  
generate exam sheet

step 6:  
update relevant cheat sheets

never stop at lecture summaries.

---

# concept note template

every concept note must contain:

## definition

clear and concise

## intuition

simple explanation

## why it exists

problem it solves

## applications

real-world uses

## advantages

## limitations

## misconceptions

## related concepts

use wikilinks

## examples

## exam relevance

high  
medium  
low

## project relevance

how it may be useful in practice

---

# linking rules

always create wikilinks.

prefer:

[[gradient descent]]

instead of plain text.

every note should connect to related concepts whenever possible.

optimize for knowledge networks.

not isolated notes.

---

# exam question generation

for each lecture generate:

5 short-answer questions

5 application questions

5 comparison questions

2 synthesis questions

prioritize likely exam patterns.

---

# answer frameworks

when appropriate generate:

- comparison frameworks
    
- evaluation frameworks
    
- design frameworks
    
- critique frameworks
    

focus on reusable thinking structures.

---

# quality standards

do not copy lecture wording.

compress ideas.

remove redundancy.

preserve meaning.

prefer tables when comparing concepts.

prefer bullet points over large paragraphs.

identify professor emphasis when detectable.

flag uncertainty explicitly.

never invent information.

---

# output philosophy

transform information into knowledge.

transform knowledge into understanding.

transform understanding into exam performance.

transform understanding into practical application.
